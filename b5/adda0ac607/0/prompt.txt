# PRD — Contexta MVP Knowledge Engine (Hybrid RAG)

## 1. Overview

Contexta’s MVP Knowledge Engine enables users to ask product management–related questions and receive grounded, cited answers derived from curated long-form content (starting with podcast transcripts).

The system uses a **Hybrid RAG architecture**:
- Semantic retrieval (pgvector)
- Keyword retrieval (Postgres full-text search)
- Merged + reranked context
- LLM answer generation with citations

The MVP prioritizes:
- Production simplicity (Node/TS monolith)
- Single database (Supabase Postgres)
- Clear source attribution
- Fast iteration

---

## 2. Problem Statement

Product managers consume knowledge across podcasts, newsletters, and blogs. Insights are:
- Hard to recall
- Hard to compare
- Hard to apply to specific challenges

Contexta should:
- Surface relevant insights quickly
- Ground answers in trusted sources
- Maintain traceability (episode, guest, timestamp)

---

## 3. Goals & Non-Goals

### Goals
- Ingest podcast transcripts into structured storage
- Enable hybrid retrieval (vector + keyword)
- Generate cited answers under 5 seconds
- Support incremental ingestion
- Keep architecture simple and extensible

### Non-Goals (MVP)
- No agent orchestration
- No graph database
- No multi-hop reasoning engine
- No automated crawling pipeline
- No advanced analytics dashboard

---

## 4. User Stories

### External User (PM)
- As a PM, I want to ask a question about product strategy and receive a concise, cited answer.
- As a PM, I want to see which episode and speaker the insight comes from.
- As a PM, I want reliable results even when I use exact framework names.

### Internal User (Admin)
- As an admin, I want to ingest a new transcript file.
- As an admin, I want embeddings generated automatically.
- As an admin, I want ingestion to be repeatable and idempotent.

---

## 5. Architecture

### High-Level Flow

Ingestion:
Transcript → Chunk → Store → Embed → Index

Retrieval:
Question → Embed → Vector Search
           → Keyword Search
           → Merge + Rerank
           → Inject Context
           → LLM → Answer + Sources

---

## 7. Data Model

### episodes
- id (uuid)
- title
- guest
- published_at
- source_url
- description
- created_at

### transcripts
- id (uuid)
- episode_id (fk)
- raw_text
- token_count
- created_at

### chunks
- id (uuid)
- episode_id (fk)
- chunk_index
- content
- token_count
- embedding (vector)
- tsv (tsvector)
- created_at

Indexes:
- vector index on embedding (pgvector)
- GIN index on tsv (full-text)

Optional (Phase 2-ready):
- tags
- entities (lightweight extraction)

---

## 8. Ingestion Pipeline (Node Service)

### Step 1 – Upload Transcript
Input: cleaned transcript text file

### Step 2 – Chunking
- Chunk size: 500–800 tokens
- Overlap: 50–100 tokens
- Store in `chunks`

### Step 3 – Embedding
- Call embedding API
- Store vector in `embedding` column

### Step 4 – Keyword Index
- Generate tsvector from content
- Store in `tsv` column

### Step 5 – Commit Transaction
Ingestion is atomic.

Performance target:
- < 2 minutes per 1-hour transcript

---

## 9. Retrieval Logic

### Step 1 – Embed User Question
Generate question embedding.

### Step 2 – Vector Search
Select top K chunks:
ORDER BY embedding <-> query_embedding
LIMIT 10

### Step 3 – Keyword Search
Full-text match:
WHERE tsv @@ plainto_tsquery(question)
ORDER BY ts_rank(...)
LIMIT 10

### Step 4 – Merge & Rerank
- Combine results
- Remove duplicates
- Optional simple scoring:
  score = semantic_score * 0.7 + keyword_score * 0.3
- Select top 6–8 chunks

### Step 5 – Prompt Construction
Prompt structure:
- System role
- User question
- Retrieved chunks
- Instructions to cite sources

### Step 6 – LLM Generation
Generate final answer with inline citations.

Latency target:
< 5 seconds end-to-end

---

## 10. Output Format

Response:
- Answer (structured markdown)
- Sources:
  - Episode title
  - Guest
  - Optional timestamp
  - Chunk reference ID

Example:

Answer:
"Three patterns emerge..."

Sources:
1. Episode: X — Guest: Y
2. Episode: Z — Guest: A

---

## 11. Quality Criteria

### Retrieval Quality
- Relevant chunk presence in top 5
- No hallucinated sources
- Citations map to real chunks

### System Performance
- P95 latency < 5s
- Vector query < 150ms
- Full-text query < 150ms

---
