Matching Logic

A technical overview of both AI pipelines Contexta uses — finding the right frameworks for your challenge, and surfacing relevant knowledge for each artifact.

Overview

Contexta runs three distinct AI flows. They are independent — each has its own trigger, query strategy, and output. None reuses another's intermediate results.

Flow 1  Challenge Matching
Triggered when: a user submits a challenge
  • AI understands the challenge
  • Hybrid retrieval (vector + keyword)
  • Scores and ranks knowledge chunks
  • AI selects 3–5 PM artifacts with explanations
Flow 2  Artifact Detail
Triggered when: a user opens an artifact page
  • Serves cached static artifact detail (or generates it once)
  • Hybrid retrieval (vector + keyword) to find related content cards
  • Optional: personalised pro tip for the user's challenge
Flow 3  Artifact Detection
Triggered when: a content item finishes processing (Run Ingestion)
  • Concatenates content chunks into a single context window
  • AI scans for PM frameworks and methodologies not yet in the catalog
  • Deduplicates against existing artifacts; flags near-duplicates
  • Inserts draft artifact records pre-filled with title, use case, description, and how-to steps
  • Creates admin notifications; admin reviews and activates or discards each draft

Flow 1

Challenge Matching

When you submit a challenge, Contexta runs a two-phase pipeline. Phase 1 generates a structured understanding of your challenge. Phase 2 uses that understanding to retrieve and rank knowledge base content, then selects the most relevant PM artifacts.

Phase 1 — Challenge Understanding

  1. AI summary — Your challenge description and context (role, company stage, team size, experience level) are sent to an AI model, which generates three structured fields: a concise summary, a focused problem statement, and a desired outcome statement.
  2. Persist — All three fields are saved to the database against the challenge record, ready for Phase 2.
  3. Return — The summary and challenge ID are returned to the client immediately. The results page renders with the summary visible while Phase 2 runs in the background.

Phase 2 — Artifact Recommendations

  1. Embedding + artifact fetch (parallel) — The combined text of the summary, problem statement, and desired outcome is converted into a numerical vector (an embedding). Simultaneously, the artifact catalog is fetched from the database.
  2. Dual retrieval (parallel) — Two independent searches run against the knowledge base (see Retrieval Methods below).
  3. Merge and deduplicate — Results from both searches are combined into a single candidate list. A chunk appearing in both searches is flagged as a stronger match.
  4. Scoring and ranking — Every candidate is scored across three dimensions and ranked (see Scoring Formula below). The top 8 chunks are kept.
  5. Domain-filtered artifact selection — The artifact catalog is filtered to artifacts whose domains overlap with the challenge domains (typically reducing 68 artifacts to 10–20 candidates). If no overlap exists, the full catalog is used.
  6. AI artifact selection — The top-ranked chunks (as context) and the filtered artifact list are sent to an AI model. The AI selects 3–5 artifacts by name, writes a 1–2 sentence explanation for each, and identifies the single most relevant artifact. It can only recommend artifacts from the provided list — it cannot invent new ones.

The Two Retrieval Methods

Semantic Search (vector search)

The challenge embedding is compared against the embedding stored for every content chunk using cosine similarity. This finds content that is conceptually close to the challenge — even if different words are used. For example, a challenge about "not knowing which feature to build next" will surface content about prioritisation frameworks even if the word "prioritisation" was never written.

Keyword Search (full-text search)

The raw challenge text is passed to PostgreSQL's built-in full-text search engine (tsvector/tsquery) using the english language configuration. This finds content chunks where the same words appear — useful for precise terminology, framework names, methodologies, and product jargon.

How the keyword search processes the query text:

  • Stop words are stripped — common English words that carry little meaning are discarded before matching (e.g. the, a, is, are, of, in, for, with, and, or, but… and approximately 170 others defined in PostgreSQL's English stop-word list).
  • Words are stemmed — remaining words are reduced to their root form using the Snowball English stemmer, so different inflections match each other. Examples: "prioritizing" → prioritiz, "challenges" → challeng, "strategies" → strategi. The same stemming is applied to content chunks at ingestion time.
  • Logical AND matching — a content chunk must contain all of the stemmed, non-stop-word tokens to be considered a match.
  • Ranking — matching chunks are ranked by ts_rank_cd with normalization 8, which divides the raw rank by the number of unique words in the chunk (preventing long chunks from unfairly dominating).
  • Enhanced by key concepts — the full-text index on each chunk also includes extracted key concepts (specific framework names and terminology identified during ingestion), so a chunk about "opportunity solution tree" will match keyword searches for those exact terms.
Keyword search failure is non-fatal. If the search returns no results or encounters an error, the pipeline silently falls back to vector-only results.

Scoring Formula

Each candidate chunk in the merged list receives a final score calculated across three dimensions:

Component How it is calculated Default weight
Domain score 1.0 if the content's domain(s) overlap with the challenge domain(s); 0.5 otherwise 0.3 (STRUCTURED_FIT_WEIGHT)
Semantic score Cosine similarity between the challenge embedding and the chunk embedding (0–1) 0.7 (EMBEDDING_SIMILARITY_WEIGHT)
Keyword score ts_rank_cd relevance from the full-text search, normalised (0–1) 0.3 (KEYWORD_RELEVANCE_WEIGHT)

All three weights are additive and independent — they do not need to sum to 1. Each weight is applied to its component and the results are summed into a final ranking score.

Domain scoring is a soft boost: content from other domains can still appear if its semantic or keyword score is high enough. It is never a hard filter.

Multi-Domain Support

Users can select more than one domain when submitting a challenge (Strategy, Discovery, Delivery, Growth, Leadership). Selecting multiple domains signals that the challenge spans multiple areas.

  • At least one domain is required; up to five can be selected simultaneously.
  • Domain overlap is treated as a scoring boost, not an exclusion rule — content from domains not selected can still appear if it is semantically relevant.
  • The artifact list passed to the AI selection step is filtered to artifacts whose domains match the selected challenge domains, reducing the candidate set and improving both speed and relevance.

Artifact Catalog

Every recommendation is drawn exclusively from a curated catalog of 68 PM frameworks and methodologies (sourced from Lenny's Frameworks). The AI selects from this known list — it cannot invent or suggest an artifact that is not in the catalog. This guarantees that all recommendations are well-defined, named practices.

The catalog spans five practice domains: Strategy, Discovery, Delivery, Growth, and Leadership.

Retrieval Depth (TOP_K)

The TOP_K setting controls how many top content chunks are retrieved from the knowledge base and passed to the artifact-selection AI call as context. Higher values give the AI more information to inform its selection (potentially improving quality at the cost of a slightly larger prompt). The default range is 1–20.

Flow 2

Artifact Detail

When a user opens an artifact page, Contexta runs up to three operations — two of which can be served from cache and one of which is personalised per user. These run independently of Flow 1 and use different retrieval strategies.

Three Operations at a Glance

  1. Static artifact detail — Structured content about the artifact (description, when to use it, steps, pro tip). Generated once by AI and cached permanently in the database. Subsequent loads are instant DB reads with no AI call.
  2. Knowledge cards — Up to 5 related knowledge base items (podcasts, articles, books) surfaced via hybrid retrieval (vector + keyword) against the artifact's own description. Requires one embedding AI call to generate the query vector; no generative AI call.
  3. Personalised pro tip — If the user arrived from a specific challenge, a small AI call replaces the generic pro tip with one tailored to their challenge context. Runs in parallel with the static detail render.

Operation 1 — Static Artifact Detail

The artifact detail is a structured object containing a description, typical use cases, key steps, common pitfalls, and a generic pro tip. It is generated once and cached — the vast majority of page loads never trigger an AI call for this.

Cache hit (most loads)

If the artifact row in the database already has a stored detail object, it is parsed and returned immediately. No AI call is made.

Cache miss (first load only)

  1. The artifact's metadata (name, domain, use case, source attribution) is assembled into a structured prompt.
  2. An AI model generates the detail object as structured JSON output.
  3. The result is validated against a Zod schema and returned to the user immediately.
  4. The validated detail is persisted asynchronously back to the database — so the next load is a cache hit. This write does not block the user-facing response.

Operation 2 — Knowledge Cards (Hybrid Search)

Knowledge cards surface real knowledge base content (podcast episodes, articles, books) that is related to the artifact — giving users a path to go deeper. Retrieval uses both vector and keyword search in parallel, then merges the results.

  1. Query construction — The artifact's title and use-case description are concatenated into a single query string. No user challenge text is involved.
  2. Parallel retrieval — Two searches fire simultaneously:
    • Vector search — The query string is embedded (text-embedding-3-small, 1536 dims) and the top 20 most similar content chunks are retrieved using cosine similarity (pgvector).
    • Keyword search — The same query string is passed directly to PostgreSQL full-text search (tsvector/tsquery), returning up to 20 chunks ranked by ts_rank_cd. No embedding needed — this fires immediately without waiting for the OpenAI round-trip.
  3. Merge — Vector results are placed first (ordered by cosine similarity), keyword-only results are appended after.
  4. Deduplication — Results are deduplicated by source content item: only the first (highest-ranked) chunk from each source is kept. This prevents one podcast episode with many chunks from dominating all 5 slots.
  5. Return top 5 — After deduplication, the top 5 unique source items are returned as knowledge cards.
There is no domain filtering — the entire knowledge base is searched regardless of the artifact's domain. Keyword search failure is non-fatal: if it returns no results, vector results alone are used. KEYWORD_RELEVANCE_WEIGHT does not apply here — there is no weighted scoring step, results are ordered by merge position (vector first, keyword additions after).

Why hybrid here?

The artifact title is a named framework (e.g. "RICE Scoring", "Opportunity Solution Tree"). Keyword search is highly precise for exact framework names — it can surface chunks that mention the artifact by name even when the cosine similarity is moderate. Vector search handles conceptual relevance. Together they cover both exact-name matches and semantically adjacent content.

Operation 3 — Personalised Pro Tip

If the user arrives at an artifact page from a challenge result (a challengeSummary is present in the request), a small AI call generates a pro tip tailored to their specific situation.

  1. The artifact metadata and the user's challenge summary (and selected domains) are assembled into a focused prompt.
  2. An AI model generates a 1–2 sentence pro tip that connects the artifact to the user's specific challenge.
  3. If this call succeeds, the personalised pro tip replaces the generic one from the static detail. If it fails, the static pro tip is shown as a fallback — failure is silent.
This is the only operation in Flow 2 that is re-run on every page load (when challenge context is present). The static detail and knowledge cards are either cached or stable — only the pro tip is personalised per visit.

Flow 3

Artifact Detection

After a content item finishes processing (chunking + embeddings), Contexta automatically scans the new chunks for PM frameworks, methodologies, and tools that are not yet in the artifact catalog. Any discoveries are inserted as draft records for admin review — the pipeline never publishes artifacts directly.

This flow is non-fatal and non-blocking: if detection fails or returns nothing, the ingestion pipeline still completes successfully. Detection can also be re-triggered manually from the content edit page ("Re-detect artifacts").

Pipeline Steps

  1. Load chunks — All content chunks for the just-processed content item are fetched from the database and concatenated into a single context string. Up to approximately 12 000 characters are used (truncated if longer).
  2. Load existing artifacts — The full list of artifact titles and slugs currently in the catalog is fetched. This list is passed to the LLM as the deduplication reference — the model is instructed not to return anything already on this list.
  3. Single LLM call — detect and generate — One AI call serves both purposes: it identifies new artifacts and generates their full content in a single structured JSON response. The prompt instructs the model to:
    • Return only artifacts that are explained or demonstrated in depth (not merely mentioned).
    • Perform a fuzzy semantic dedup against the existing list — not an exact string match.
    • Flag borderline cases as is_possible_duplicate = true and populate possible_duplicate_of with the slug of the closest existing artifact.
    • For each new artifact, generate: title, slug (kebab-case), domains (1–3), use case (1–2 sentences), description (3–5 sentences), how-to intro (1–2 sentences), and 3–8 how-to steps.
    • Return an empty array if no new artifacts are found.
  4. Schema validation — The LLM response is parsed as JSON and validated against a Zod schema. Any response that fails validation is discarded (non-fatal).
  5. Slug dedup — Before inserting each detected artifact, its slug is checked against the in-memory set of known slugs. An artifact whose slug already exists is skipped silently. This is a cheap second guard after the LLM-level dedup.
  6. Insert draft artifacts — Each genuinely new artifact is inserted into the artifacts table with:
    • status = 'draft' — never published automatically.
    • is_ai_generated = true
    • source_content_id pointing to the content item that triggered detection.
    • possible_duplicate_of populated if the LLM flagged it as borderline.
    • detail JSONB containing description, how-to intro, and how-to steps.
  7. Admin notifications — For each inserted artifact, an admin_notifications row is created (type = 'artifact_detected') with a direct link to the artifact editor. The bell icon in the admin panel shows the unread count and a toast appears via Supabase Realtime.

Deduplication Strategy

Two independent dedup guards run in sequence:

Guard Mechanism What it catches
LLM semantic dedup Existing artifact titles passed as context; LLM performs fuzzy matching on meaning Near-synonyms, alternate names, rebranded variants (e.g. "Jobs to Be Done" vs "JTBD")
Slug exact dedup Set membership check on known slugs before each insert Exact slug collisions — fast and free, runs in memory

Artifacts that pass both guards but are semantically close to an existing one are inserted with possible_duplicate_of set. The artifact editor displays a warning banner prompting the admin to review before activating.

Context Window Budget

The chunks fed to the LLM are capped at ~12 000 characters. For most content items this covers the full transcript; for very long pieces (e.g. a full book transcript) only the first 12 K characters are used. This keeps the prompt within a predictable token budget while still capturing the densest material (which typically appears early in the chunk sequence).

Admin Workflow After Detection

Detection creates raw material — admin review determines what reaches the catalog. The admin flow is:

  1. Bell icon shows unread notification count; toast fires on arrival via Realtime.
  2. Admin opens /admin/artifacts and filters to Draft status.
  3. For each draft, the editor shows all generated fields plus a read-only info panel indicating the source content item and any duplicate warning.
  4. Admin can edit any field, change status to Pending review or Active, or delete the draft.
  5. Setting status to Active triggers a second AI flow — News Proposal Generation — which creates a draft news card for the Journey feed.

News Proposal Generation

Whenever a content item or artifact transitions to status = 'active' via the admin API, a separate lightweight AI call generates a draft news post for the Contexta Journey feed. This is also non-fatal — activation succeeds even if proposal generation fails.

  1. Source metadata assembled — For content: title, author, source type (podcast / video / article / book), domains. For artifacts: title, domains, use case, description.
  2. Single LLM call — The model is prompted to write an engaging news card in the tone of a professional PM newsletter. It returns: type (podcast / artifact / article), title (max 80 chars), description (max 200 chars, 1–2 sentences), and published_date (display string, e.g. "Mar 2026").
  3. Insert draft news post — A news_posts row is created with status = 'draft', is_ai_generated = true, and a source_id back-reference to the originating content item or artifact.
  4. Admin notification — An admin_notifications row is created (type = 'news_proposal_generated') with a link to the news post editor. The news list shows an AI draft badge on auto-generated posts; they are edited and published identically to manual posts.
Every activation triggers a new proposal — including re-activations. This is intentional: a content item updated and re-activated might warrant a fresh news card.
