Matching Logic

How Contexta finds the right PM frameworks for your challenge — a technical overview of the retrieval and scoring pipeline.

Overview

When you submit a challenge, Contexta runs a two-phase pipeline. Phase 1 generates a structured understanding of your challenge using an AI model. Phase 2 uses that understanding to search the knowledge base and select the most relevant PM artifacts from a curated catalog.

Phase 1 — Challenge Understanding

  1. AI summary — Your description and context (role, company stage, team size, experience level) are sent to an AI model, which generates a concise summary, a focused problem statement, and a desired outcome statement.
  2. Persist — All three fields are saved to the database against the challenge record, ready for Phase 2.
  3. Return — The summary and challenge ID are returned to the client immediately. The results page renders with the summary visible while Phase 2 runs in the background.

Phase 2 — Artifact Recommendations

  1. Embedding + artifact fetch (parallel) — The combined text of the summary, problem statement, and desired outcome is converted into a numerical vector (an embedding). Simultaneously, the artifact catalog is fetched from the database.
  2. Dual retrieval — Two independent searches run in parallel against the knowledge base (see below).
  3. Merge and deduplicate — Results from both searches are combined into a single candidate list. A chunk appearing in both searches is flagged as a stronger match.
  4. Scoring and ranking — Every candidate is scored across three dimensions and ranked (see Scoring Formula below).
  5. Domain-filtered artifact selection — The artifact catalog is filtered to artifacts whose domains overlap with the challenge domains. This typically reduces the catalog from 68 artifacts to 10–20 candidates, making the AI call faster and more focused. If no overlap exists, the full catalog is used.
  6. Artifact selection pass — The top-ranked content chunks (as context) and the filtered artifact list are sent to an AI model. The AI selects 3–5 artifacts by name, writes a 1–2 sentence explanation for each, and identifies the single most relevant artifact. It can only recommend artifacts from the provided list — it cannot invent new ones.

The Two Retrieval Methods

Semantic Search (vector search)

The challenge embedding is compared against the embedding stored for every content chunk using cosine similarity. This finds content that is conceptually close to the challenge — even if different words are used. For example, a challenge about "not knowing which feature to build next" will surface content about prioritisation frameworks even if the word "prioritisation" was never written.

Keyword Search (full-text search)

The raw challenge text is passed to PostgreSQL's built-in full-text search engine (tsvector/tsquery) using the english language configuration. This finds content chunks where the same words appear — useful for precise terminology, framework names, methodologies, and product jargon.

How the keyword search processes the query text:

  • Stop words are stripped — common English words that carry little meaning are discarded before matching (e.g. the, a, is, are, of, in, for, with, and, or, but… and approximately 170 others defined in PostgreSQL's English stop-word list).
  • Words are stemmed — remaining words are reduced to their root form using the Snowball English stemmer, so different inflections match each other. Examples: "prioritizing" → prioritiz, "challenges" → challeng, "strategies" → strategi. The same stemming is applied to content chunks at ingestion time.
  • Logical AND matching — a content chunk must contain all of the stemmed, non-stop-word tokens to be considered a match.
  • Ranking — matching chunks are ranked by ts_rank_cd with normalization 8, which divides the raw rank by the number of unique words in the chunk (preventing long chunks from unfairly dominating).
  • Enhanced by key concepts — the full-text index on each chunk also includes extracted key concepts (specific framework names and terminology identified during ingestion), so a chunk about "opportunity solution tree" will match keyword searches for those exact terms.
Keyword search failure is non-fatal. If the search returns no results or encounters an error, the pipeline silently falls back to vector-only results.

Scoring Formula

Each candidate chunk in the merged list receives a final score calculated across three dimensions:

Component How it is calculated Default weight
Domain score 1.0 if the content's domain(s) overlap with the challenge domain(s); 0.5 otherwise 0.3
Semantic score Cosine similarity between the challenge embedding and the chunk embedding (0–1) 0.7
Keyword score ts_rank_cd relevance from the full-text search, normalised (0–1) 0.3

All three weights are additive and independent — they do not need to sum to 1. Each weight is applied to its component and the results are summed into a final ranking score.

Domain scoring is a soft boost: content from other domains can still appear if its semantic or keyword score is high enough. It is never a hard filter.

Multi-Domain Support

Users can select more than one domain when submitting a challenge (Strategy, Discovery, Delivery, Growth, Leadership). Selecting multiple domains signals that the challenge spans multiple areas.

  • At least one domain is required; up to five can be selected simultaneously.
  • Domain overlap is treated as a scoring boost, not an exclusion rule — content from domains not selected can still appear if it is semantically relevant.
  • The artifact list passed to the AI selection step is filtered to artifacts whose domains match the selected challenge domains, reducing the candidate set and improving both speed and relevance.

Artifact Catalog

Every recommendation is drawn exclusively from a curated catalog of 68 PM frameworks and methodologies (sourced from Lenny's Frameworks). The AI selects from this known list — it cannot invent or suggest an artifact that is not in the catalog. This guarantees that all recommendations are well-defined, named practices.

The catalog spans five practice domains: Strategy, Discovery, Delivery, Growth, and Leadership.

Retrieval Depth (TOP_K)

The TOP_K setting controls how many top content chunks are retrieved from the knowledge base and passed to the artifact-selection AI call as context. Higher values give the AI more information to inform its selection (potentially improving quality at the cost of a slightly larger prompt). The default range is 1–20.
